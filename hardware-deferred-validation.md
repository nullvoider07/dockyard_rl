# Hardware-deferred validation ledger

**Validation of `dockyard_rl` is incomplete.** It is developed **without access to a GPU
or a multi-node cluster**, so a large part of the system has never been run on the
hardware it targets. Everything that can be checked statically or numerically on CPU
is â€” every change ships with `py_compile`, a clean canonical `pyright` pass, and CPU
unit/parity tests (model forward-logit parity against the HuggingFace reference,
loss/gradient parity against the torch reference, name-map round-trips, config
resolution, and byte-identity of the unchanged path when a feature is disabled). But the
checks that require real hardware **have not been performed**, because the resources to
run them are not available at the developer's end.

This ledger is the **honest record of that gap** â€” the validations that remain
**pending** until GPUs/a cluster are available: live NCCL/collective numerics, multi-GPU
sharding and device-mesh composition, CUDA-only kernels (`torch._grouped_mm`,
`ragged_all_to_all`), bf16 numerics, weight refit into a running inference engine, and
end-to-end training convergence. Each item below names the exact assumption that is
written to be correct but is **not yet verified**, why it cannot be tested here, and the
concrete check that would close it on hardware. Until that check is run, the item should
be treated as **unproven** â€” it may pass, or it may surface a real bug.

I publish this rather than imply full validation. If you have GPU/cluster resources and
run any of these pending checks, results (pass or fail) are very welcome â€” open an issue
or PR referencing the `HV-` id.

**How to read a row.** Each row records the area + source file, the unverified
assumption, a severity, and the bring-up check that closes it. Severity:
**High** = silent wrong training / crash if the assumption is wrong;
**Med** = degraded behavior or a loud failure that's easy to spot; **Low** = cosmetic /
perf only.

**Subsystems covered.** Torch MoE / expert-parallelism (HV-1â€“3, 8â€“15), weight-sync &
serving (HV-4, 7, 16), the async data plane (HV-17â€“18), rollouts & structured tool-use
(HV-19â€“23), preference/offline algorithms (HV-24â€“26), and the JAX trainer re-platform
(HV-27â€“33). The standing path to closing most of these is a single-node un-packed
bring-up on a tiny model, then a small MoE model for the EP-specific rows.

## Notes
- **Chunked delta-rule kernel (perf, not correctness):** the J11 linear-attention recurrence uses a `lax.scan` (O(S) sequential); the HF chunked kernel is a perf optimization computing identical math (CPU-parity-proven). Implementing/benchmarking the chunked form is a J10/perf task (any GPU), not a correctness HV â€” correctness is already covered.
- **`apply_mask_to_padding_states` omitted in the JAX linear-attn module (right-padding assumption):** correct for right-padded causal training (padded positions never pollute real ones); would bite only under left-padding. A documented assumption, not a GPU-gated check â€” the trainer must keep right-padding.
- Resolved-without-hardware (NOT deferred): A.1 #2569 reasoning-parser wiring was hardened
  and verified against a real installed vLLM (0.21, same parser-API line as the 0.22 pin);
  A.3 #2686/#2478 numerics are bit-validated against the original in pure torch.
- The standing path to closing most of these is the single-node un-packed bring-up the
  production-readiness audit calls for (tiny model, packing off, `_v2=false` for the dense
  v1 path), then a small MoE model (Qwen3-MoE / GLM) for the EP-specific items (HV-1/2/8).
